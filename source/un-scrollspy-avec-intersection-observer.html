Un quoi ?

Un scrollspy est un pattern qui traque la position de l'utilisateur dans un conteneur, généralement pour mettre à jour l'interface. L'exemple le plus commun est le curseur de la barre de défilement qui indique visuellement cette position.

La plupart du temps, ce pattern est utilisé pour mettre en évidence un lien vers la section consultée au sein d'une navigation interne. Vous pouvez observer ce comportement via la navigation présente sur cette page.

Avec quoi ?

IntersectionObserver est une API déjà bien supportée qui permet de suivre de manière asynchrone l'intersection d'éléments avec un parent. Plus précisément, une fonction de callback est appelée lorsque leur ratio de visibilité passe une ou plusieurs valeurs données.

On peut donc s'attendre à de meilleures performances qu'en utilisant l'événement scroll car non seulement le callback ne s'exécute jamais pour rien (le code le plus rapide est celui qui ne s'exécute pas), mais on a plus à tester la position des éléments nous-mêmes. Cela peut sembler trivial mais permet de se prémunir contre le forced synchronous layout (remplacez layout par reflow si vous êtes sous Firefox).

Le quoi ?

Lorsqu'un navigateur doit mettre à jour l'affichage d'une page web, il génère le rendu dans un ordre bien précis :

  1. JavaScript
  2. Calculs de style
  3. Mise en page
  4. Rendu
  5. Composition

Vous pouvez trouver la description de ces étapes dans l'article Performance de rendu des Web Fundamentals.

Lors de son exécution, JavaScript dispose des informations de mise en page du rendu précédent, mais il peut également la modifier. Dans ce cas, les informations deviennent périmées et leur lecture forcerait le navigateur à les recalculer, ralentissant d'autant l'exécution du script. Lorsque ce cycle est répété (typiquement, dans une boucle) on parle de layout thrashing. À éviter.

Comme IntersectionObserver permet de s'affranchir de la lecture des informations de mise en page, l'utiliser vous prémunit contre le forced synchronous layout.
